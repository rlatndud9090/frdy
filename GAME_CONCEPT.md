# 게임 컨셉: 마왕의 은밀한 조력

## 핵심 스토리

**주인공**: 마왕 (환생한 일반인)

**배경 설정**:
- 평범한 일반인이 마왕으로 환생함
- 원래 세계로 돌아가기 위한 유일한 방법: 용사의 손에 죽는 것
- 문제: 부하들이 충성심이 너무 강해서 용사를 막으려고 함

**게임 목표**:
- 부하들 **몰래** 용사를 도와야 함
- 용사가 마왕성을 성공적으로 돌파하도록 만들기
- 들키지 않으면서 용사의 승리를 보장해야 함

## 게임 메커니즘

### 1. 맵 구조 (절차적 생성)

**마왕성의 혼돈 마법**:
- 문을 열 때마다 구조가 뒤바뀌는 혼돈의 마법이 걸려있음
- 로그라이크 특성을 위한 절차적 생성 구현

**맵 구조**:
- 슬레이 더 스파이어 스타일의 **Node-Edge Graph** 구조
- 여러 **'층'**으로 구성 (슬더스의 '막'과 동일한 역할)
- 각 층은 연결된 노드들로 구성됨

**Node 타입**:
- 🗡️ **전투 노드**: 적과의 전투 발생
- ❓ **이벤트 노드**: 특수 이벤트 발생

### 2. 전투 시스템

**기본 구조**:
- 턴제 전투
- 용사와 적 각각 다음 속성 보유:
  - 공격력
  - 체력
  - 행동 패턴

**용사 성장**:
- 전투 승리 시 강화
- 이벤트를 통한 강화
- 공격력, 체력, 행동 패턴이 점진적으로 향상

### 3. 마안(魔眼) - 전투 예측 시스템

마왕이 가진 고유 능력 **'마안'**은 미래의 전투 흐름을 미리 내다보는 눈이다.

**핵심 컨셉**:
- 마안의 힘으로 용사와 적의 **모든 턴의 행동을 미리 예측**하여 볼 수 있음
- 전투 시작 시 양측의 행동 패턴을 기반으로 **전체 전투의 예측 타임라인**이 생성됨
- 플레이어(마왕)는 이 예측을 보고 **어디에 개입할지** 전략을 세움

**예측 타임라인 UI (Prediction Timeline)**:
- 화면 중앙 상단에 **네모 박스 배열(Array)**로 표시
- 용사/적 캐릭터 위치와 상단 게이지 바 사이 공간에 배치
- 각 박스는 하나의 **예측된 행동(Action)**을 나타냄
- 박스 색상/아이콘으로 행동 주체(용사/적)와 행동 타입(공격/방어/스킬)을 구분
- 예시 레이아웃:
```
[의심 게이지]              [마력 게이지]

  [용사공격] [적1공격] [적2방어] [용사공격] [적1공격] ...
       ↑ 예측 타임라인 (가로 스크롤 가능)

[용사]                      [적 1]
[HP 게이지]                 [HP 게이지]
                            [적 2]
                            [HP 게이지]

[카드1] [카드2] [카드3] [패스] [턴종료]
```

**마왕 개입 카드의 타임라인 삽입**:
- 카드를 사용할 때, 타임라인의 **어느 위치에 끼워넣을지** 플레이어가 결정
- 카드를 타임라인 박스 사이로 드래그하여 삽입 위치 선택
- 삽입된 카드는 해당 시점에 실행됨 (예: 적의 공격 직전에 방어 버프 삽입)
- 삽입 후 타임라인이 실시간으로 재계산되어 결과 예측이 갱신됨

**카드 타입별 타임라인 상호작용**:

| 카드 타입 | 타임라인 상호작용 | 예시 |
|-----------|-------------------|------|
| **삽입형 (Insert)** | 타임라인 특정 위치에 끼워넣음 | 방어 버프, 힐, 공격 카드 |
| **전역형 (Global)** | 위치 무관, 전체 전투에 적용 | 전체 공격력 변동, 환경 효과 |
| **조작형 (Manipulate)** | 기존 행동의 순서를 바꾸거나 제거 | 행동 순서 교환, 행동 무효화, 행동 지연 |

**조작형 카드 세부**:
- **순서 교환**: 타임라인에서 두 행동의 순서를 바꿈 (예: 적 공격과 용사 공격 순서 교환)
- **행동 제거**: 특정 행동을 타임라인에서 제거 (예: 적의 강력한 공격을 무효화)
- **행동 지연**: 특정 행동을 타임라인 뒤쪽으로 밀어냄

### 4. 마왕의 개입 시스템

**행동 카드 사용**:
- 매 턴마다 **'마력'**을 소모하여 **'행동 카드'** 사용
- 행동 카드로 용사를 **조력** 또는 **방해** 가능
- **마안의 예측 타임라인**을 보며 최적의 개입 시점을 선택하는 것이 핵심 전략

**마왕의 성장**:
- 용사가 전투를 승리할 때마다 새로운 **행동 카드** 획득
- 환생 컨셉: 처음엔 약한 힘만 사용 가능 → 점점 깨달아 강해짐

### 5. 의심 시스템 (핵심 메커니즘)

**의심 수치**:
- 용사를 **도우면** → 의심 수치 **증가** ⬆️
- 용사를 **방해하면** → 의심 수치 **감소** ⬇️
- 의심 수치가 **MAX** 도달 → 부하들의 반란 → **게임 오버**

**전략적 중요성**:
- 너무 노골적으로 도우면 들킴
- 적절히 방해하면서 은밀하게 조력해야 함
- 의심받지 않는 균형 유지가 승리의 열쇠

### 6. 이벤트 및 진로 조작

**이벤트 발생 시**:
- 마왕의 선택지 제공 (조력/방해)
- 선택에 따라 의심 수치 변동

**진로 수정**:
- 용사가 갈래길에서 방향 선택 시
- 마왕이 용사의 **진로를 은밀히 수정** 가능
- 진로 수정도 의심 수치에 영향

## 기술 아키텍처

### 핵심 기술 스택

- **엔진**: Love2D (Lua)
- **플랫폼**: PC
- **OOP**: middleclass 라이브러리
- **애니메이션**: flux (tween)
- **유틸리티**: lume

### 아키텍처 원칙

- **Scene 기반 화면 관리** (State Machine + Stack)
- **EventBus를 통한 시스템 간 느슨한 결합**
- **Strategy 패턴**으로 카드 효과 확장
- **Factory 패턴**으로 맵 생성
- **Data-driven 설계** (Lua 테이블 기반 데이터 파일)

### 디렉토리 구조

```
src/
├── core/     - 프레임워크 (Game, SceneManager, EventBus)
├── map/      - 맵 시스템 (Node, Edge, Floor, MapGenerator)
├── combat/   - 전투 시스템 (Entity, Hero, Enemy, TurnManager)
├── card/     - 카드/개입 시스템 (Card, Deck, Mana, Suspicion)
├── event/    - 이벤트 시스템 (Event, Choice, EventManager)
├── scene/    - Scene 구현 (MapScene, CombatScene 등)
├── ui/       - UI 위젯 (Button, Gauge, Minimap 등)
└── anim/     - 애니메이션 (Camera, Transition)

data/         - 게임 데이터 정의 (cards, events, enemies)
assets/       - 아트/사운드 에셋
lib/          - 외부 라이브러리 (middleclass, flux, lume)
```

### 설계 원칙 (SOLID)

- **단일 책임 원칙**: 각 클래스는 하나의 책임만 가짐
- **개방-폐쇄 원칙**: 새 노드/카드 타입 추가 시 기존 코드 수정 불필요
- **의존성 역전**: EventBus로 시스템 간 결합도 최소화
